# Bloom 🌱

> A local-first, keyboard-driven note-taking app inspired by Logseq and Emacs Org-mode + Org-roam.
> Write freely. Let patterns emerge. Watch your notes bloom into knowledge.

---

## Core Philosophy

You should never have to think about where a note belongs. Keep adding notes to daily journal pages or wherever feels natural. Over time, patterns emerge — and when you're ready, create a page for a topic. That page shows a timeline of all related notes from across your files, without duplicating text. Linking is a first-class feature.

---

## Goals

### G1: Local-Only, Private by Default

- All data stays on your machine. No cloud, no server, no sync.
- Safe for company secrets, personal thoughts, and sensitive information.
- Bloom makes zero network calls by default. The only exception is the opt-in MCP server (G17), which listens on localhost for local LLM integration. Even then, Bloom itself never contacts the internet — the user's LLM client is their own responsibility.

### G2: Markdown with Bloom Extensions

- Notes are stored as Markdown files on disk — portable, readable, version-controllable.
- Bloom extends Markdown with linking, block IDs, and tags.
- File format is abstracted behind a `DocumentParser` trait — swap to Org-mode or other formats later.

### G3: UUID-Based Stable Linking

- Every page has a UUID set at creation (8-char hex in frontmatter). On collision, regenerate.
- Sections and blocks get UUIDs lazily — only when first linked to.
- Links use UUID + display hint: `[[8f3a1b2c|Text Editor Theory]]`.
- Renames never break links. The UUID is permanent; display hints are cosmetic and auto-updated.
- **Frontmatter `title` is the source of truth.** Filenames are derived from it via sanitization: characters invalid on any OS (`/ \ : * ? " < > |`) are replaced with `-`, leading/trailing whitespace is trimmed, and filenames are capped at 200 characters (truncated with a hash suffix if longer). If an external tool renames the file, Bloom reverts the filename to match the frontmatter title. Renaming a page is done by editing the frontmatter title in Bloom — the filename follows automatically.
- **Display hints are eagerly updated in the background.** When a page is renamed, a background thread scans all files and updates stale `[[uuid|old hint]]` to `[[uuid|new hint]]`. For files with open buffers, the update is applied directly to the rope via the same channel as MCP/UI edits (cursor-safe, undo-able). For files not open, the background thread writes to disk via the disk writer channel. Completes in under a second for typical vaults.
- **Case collision detection**: On case-insensitive filesystems (macOS, Windows), Bloom detects when a new page title would collide with an existing filename (e.g., "Rust" vs "rust") and appends the UUID suffix to disambiguate: `Rust-8f3a1b2c.md`.

### G4: Linking as a First-Class Feature

**Link creation UX:** Typing `[[` in the editor triggers a fuzzy picker inline. Search for a page by title, select it, and Bloom inserts `[[uuid|title]]` for you. The user never types or sees UUIDs directly.

**Page links:**

| Syntax | Purpose |
|--------|---------|
| `[[page-id\|display]]` | Link to page |
| `[[page-id^section-id\|display]]` | Deep link to heading |
| `[[page-id^block-id\|display]]` | Deep link to block |

**Block IDs:**

```markdown
- Ropes are O(log n) for inserts ^rope-perf
```

Append `^id` to any line to make it referenceable. IDs are auto-generated by Bloom when first linked, or set manually.

**Tags:** `#tag` inline or in frontmatter `tags: [...]`. Parsing rules: tags must start with a Unicode letter (`#rust` ✅, `#プログラミング` ✅, `#123` ❌), must be preceded by whitespace or start of line (`foo#bar` is not a tag). All Bloom extensions — `[[links]]`, `^block-ids`, `#tags`, and `@timestamps` — are ignored inside code spans, fenced code blocks, and YAML frontmatter.

**Timestamps** (Org-mode inspired):

| Syntax | Purpose | Example |
|--------|---------|---------|
| `@due(2026-03-05)` | Deadline — when it must be done | `- [ ] Submit report @due(2026-03-05)` |
| `@start(2026-03-03)` | Scheduled start — when you plan to begin | `- [ ] Read CRDT paper @start(2026-03-03)` |
| `@at(2026-03-02 14:00)` | Event — specific date/time | `Meeting with Priya @at(2026-03-02 14:00)` |

Timestamps are optional. Time portion (`14:00`) is optional. Used by the Agenda view (G15) to build Overdue / Today / Upcoming sections.

### G5: Unlinked Mentions (Discovery)

- Bloom scans all notes for text matching page titles.
- Matches appear as "Unlinked Mentions" on each page — surfacing connections you haven't made explicit yet.
- Promote to an explicit `[[link]]` or ignore. You decide. **Batch promote**: from the unlinked mentions picker (`SPC s u`), mark multiple results with `Tab`, then `Enter` to promote all at once.
- Explicit links = high-signal timeline. Unlinked mentions = discovery zone.

### G6: Timeline View

When viewing a page, `SPC l t` opens a timeline panel showing a chronological list of all notes that link to it — from journal entries or any other file. Notes appear embedded in the timeline without physical duplication. The timeline can be pinned as a persistent window (G11) alongside the editor. This is Bloom's signature feature.

### G7: Vim-Like Modal Editing

- Full Vim grammar: `[count] [operator] [motion/text-object]`.
- Modes: Normal, Insert, Visual, Command.
- Pending keystrokes shown in the status bar.
- State machine implementation with composable operators, motions, and text objects.
- A non-negotiable core feature.

### G8: Which-Key Style Command Discoverability

- Leader key (`SPC` in Normal mode) opens a hierarchical command tree.
- After a brief timeout (~300ms), a popup shows all available next keys with descriptions.
- Categories: `f` (file), `s` (search), `l` (links), `j` (journal), `t` (tags), etc.
- Works during Vim grammar too — pressing `d` and pausing shows available motions.
- `SPC ?` opens a fuzzy-searchable list of all commands.

### G9: Undo Tree

- Full branching undo history, not linear undo/redo.
- Every edit creates a node in the tree. Undoing and making a new edit creates a branch.
- Navigate the full tree of document states within a session.
- **Persistent** — the undo tree is serialized to SQLite on quit and restored on next launch (same model as VS Code). Pruned when the buffer is closed or after 24 hours. For long-term history beyond the undo tree, Bloom's git-based time travel provides session-level snapshots (see `docs/HISTORY.md`).

### G10: Cross-Platform — macOS and Windows

- macOS and Windows support are non-negotiable.
- Platform-native keyboard shortcuts (Cmd on macOS, Ctrl on Windows) for common actions.
- Vim keybindings are identical across platforms.
- Platform shortcuts take priority over Vim layer (Cmd+S always saves, even in Normal mode).

### G11: Window Management (Doom Emacs-style)

Split views for side-by-side editing, navigated entirely by keyboard.

| Trigger | Action |
|---------|--------|
| `SPC w v` | Vertical split |
| `SPC w s` | Horizontal split |
| `SPC w h/j/k/l` | Navigate between windows |
| `SPC w d` | Close window |
| `SPC w =` | Balance window sizes |
| `SPC w m` | Maximize/restore current window (toggle) |

Each window is an independent view — can show different pages, a timeline alongside an editor, the agenda next to a journal, or a pinned backlinks panel.

### G12: Structured Search Filters

The fuzzy picker (`SPC s s`) supports composable filters for structured queries — no separate query language needed.

- While in the picker, press modifier keys to add/remove filters:
  - `Ctrl+T` — filter by tag
  - `Ctrl+D` — filter by date range
  - `Ctrl+L` — filter by "links to" a specific page
  - `Ctrl+S` — filter by task status (open/done)
- Filters stack: fuzzy text + tag + date range can all be active simultaneously.
- Active filters shown as pills below the search input. Remove with `Backspace` on the pill.
- This replaces a dedicated query DSL — same power, nothing new to learn.

### G13: Logseq Import

One-way, non-destructive import from Logseq. The importer reads from the Logseq directory and writes to the Bloom vault — **Logseq files are never modified**.

**Syntax mapping:**

| Logseq Syntax | Bloom Equivalent | Notes |
|---------------|-----------------|-------|
| `[[page link]]` | `[[uuid\|page link]]` | UUID generated, title preserved |
| `((block-ref))` | `[[page-uuid^block-id\|text]]` | Logseq `id::` properties mapped to `^block-id` anchors |
| `{{embed [[page]]}}` | `[[uuid\|page]]` | Converted to Bloom link (transclusion deferred to post-v1) |
| `{{embed ((block))}}` | `[[page-uuid^block-id\|text]]` | Converted to block link (transclusion deferred to post-v1) |
| `/namespace/page` | Page titled "Namespace - Page" with auto-tags `#namespace` | Hierarchy flattened to title + tags |
| `TODO item` | `- [ ] item` | |
| `DONE item` | `- [x] item` | |
| `DOING` / `LATER` / `NOW` / `WAITING` / `CANCELLED` | `- [ ] item` with `@status(doing)` comment | Best-effort mapping; original keyword preserved in a comment |
| `key:: value` (properties) | Preserved as arbitrary YAML frontmatter | Bloom ignores unknown keys but preserves them |
| `tags:: a, b` | `tags: [a, b]` in frontmatter | |
| `alias:: Name` | Preserved as `aliases: [Name]` in frontmatter | No runtime alias resolution in v1; preserved for future use |
| `#+BEGIN_QUOTE` | `> blockquote` | Org-mode blocks converted to Markdown equivalents |
| Advanced queries `{{query ...}}` | Preserved as code blocks with a `⚠️ Logseq query` comment | No Bloom equivalent; not silently dropped |

**Directory mapping:**

| Logseq | Bloom |
|--------|-------|
| `journals/` | `journal/` (filenames: `2026_02_28.md` → `2026-02-28.md`) |
| `pages/` | `pages/` |
| `assets/` | `images/` (non-image assets placed in `images/` too) |

**Import process:**

1. User triggers import via setup wizard (G21) or `:import-logseq` command.
2. Bloom reads the Logseq directory (never modifies it).
3. Files are copied and transformed into the Bloom vault.
4. **Import report** is generated and displayed: pages imported, links resolved, warnings (unresolved refs, dropped content), errors (unparseable files).
5. User reviews the report. Failed files can be re-imported individually.

**Safety guarantees:**

- Logseq files are never modified. Import is a copy + transform.
- If import fails midway, the partially-imported files remain in the vault (valid Bloom pages). Re-running import skips already-imported pages (matched by title).
- User is advised to back up both Logseq and Bloom directories before import.

### G14: Daily Journal

The journal is Bloom's default writing surface — the place you go when you don't know where something belongs.

- Journal pages live in a `journal/` directory, one file per day: `journal/2026-02-28.md`.
- `SPC j j` opens today's journal. A buffer is created in memory immediately, but the file is written lazily (only when auto-save triggers after the first edit).
- `SPC j p` / `SPC j n` navigate to previous / next day's journal.
- Journal pages are regular Bloom pages — they support all linking, tagging, and block ID features.
- There is no separate "inbox." The journal is your inbox.
- **Quick capture**: `SPC j a` (journal → append) opens a single-line input at the bottom of the screen. Type a thought, press `Enter`, and it's appended to today's journal without leaving the current buffer. For tasks: `SPC j t` appends a `- [ ]` prefixed line.

### G15: Agenda View (Org-mode inspired)

An aggregated view of tasks and scheduled items across all pages, inspired by Org-mode's agenda.

- `SPC a a` opens the agenda view as a full-screen takeover.
- Shows all unchecked `- [ ]` items and `@at` events from all pages.
- Grouping uses timestamps: `@due` in the past = overdue, `@due` or `@start` today = today, future = upcoming. Tasks without timestamps grouped by source page date.
- Filter by tag, page, or date range.
- Act on tasks directly from the agenda: check off, reschedule (move to another day's journal), jump to source.

See [WINDOW_LAYOUTS.md](WINDOW_LAYOUTS.md#agenda-view) for the detailed wireframe and column layout.

### G16: Fuzzy Picker (Doom Emacs-style Narrowing)

A universal, keyboard-driven fuzzy picker for navigating everything in Bloom. Inspired by Doom Emacs's Vertico + Orderless + Consult + Marginalia stack.

**Orderless fuzzy matching:** Type fragments in any order. `edt thry` matches "**Ed**i**t**or **Th**eo**ry**". Results narrow in real-time.

**Picker surfaces:**

| Trigger | Searches | Metadata |
|---------|----------|----------|
| `SPC f f` | All pages | tags, date |
| `SPC b b` | Open buffers | modified indicator |
| `SPC s s` | Full-text across all notes | page name, matching line |
| `SPC s j` | Journal entries | date |
| `SPC s t` | All tags (on select: transitions to search filtered by tag) | note count per tag |
| `SPC s l` | Backlinks to current page | source page, date |
| `SPC s u` | Unlinked mentions | source page, context |
| `SPC SPC` | All commands (like Emacs `M-x`) | keybinding, category |

**UX principles:**

- **Marginalia** — right-aligned metadata (tags, date, keybinding) for each result.
- **Preview** — preview selected item without leaving the picker.
- **Actions** — act on results beyond just "open" (e.g., delete, rename, copy link).
- **Recency bias** — recently opened pages score higher.
- **Persistent history** — remembers past selections and ranks them higher.

**Implementation:** `nucleo` crate (from Helix editor) for the fuzzy matching engine. Generic `Picker<T>` backed by a `PickerSource` trait — each surface (pages, buffers, tags, commands) is a concrete implementation.

### G17: MCP Server (LLM Integration) — Opt-In

Bloom includes a local MCP (Model Context Protocol) server, allowing any LLM client to read, search, and write to your notes. **Disabled by default** — the user explicitly enables it and Bloom shows a clear notice that notes may be sent to the configured LLM client.

The MCP server is a **virtual editor** — it edits the same rope buffer as the UI, never writes to disk directly. This means:

- **No conflicts**: MCP and the UI share one rope per file. One source of truth.
- **Undo includes MCP edits**: LLM changes are nodes in the undo tree. User can undo them.
- **Live UI updates**: MCP edits appear in the UI in real-time.
- **Background buffers**: If MCP edits a file not open in the UI, Bloom loads it into a background buffer. Auto-save writes it to disk. Background buffers are evicted from memory after 60 seconds of inactivity (no edits) once saved to disk. This prevents memory leaks from bulk MCP operations.
- **Notification**: Status bar shows `"MCP: added N lines to page.md"` when an edit occurs.

When disabled, Bloom makes zero network calls.

**Security configuration** (`config.toml`):

```toml
[mcp]
enabled = false              # must be explicitly enabled
mode = "read-write"          # "read-only" or "read-write"
exclude_paths = ["journal/therapy/*", "pages/Passwords.md"]  # glob patterns
```

- **Read-only mode**: Only `search_notes`, `read_note`, `list_pages`, `list_tags`, `get_backlinks`, `get_agenda`, `list_templates`, and `resolve_journal` are available. All write tools return permission errors.
- **Path exclusion**: Excluded files are invisible to all MCP tools — not returned in search, not readable, not editable.

**Exposed tools (all title-based — LLM never sees UUIDs):**

| Tool | Parameters | Edge case behavior |
|------|-----------|-------------------|
| `search_notes` | `query`, `filters?` | Returns empty list if no matches |
| `read_note` | `title` (fuzzy matched) | Error if no match; returns top-N candidates if ambiguous. Returns raw Markdown with display hints (not UUIDs). |
| `create_note` | `title`, `content`, `tags[]`, `template?`, `template_values?` | If title exists, appends to end of existing page |
| `edit_note` | `title`, `old_text`, `new_text` | Search-and-replace: finds `old_text` in the page, replaces with `new_text`. Error if not found. Error if ambiguous (multiple matches — include more context). |
| `add_to_journal` | `content`, `date?` | Default: today. Optional ISO date for backfilling. Auto-creates journal if needed. |
| `list_pages` | `tag_filter?` | Returns empty list if no matches |
| `list_tags` | — | Returns all tags with counts |
| `list_templates` | — | Returns available template names and descriptions |
| `get_backlinks` | `title` | Returns empty list if no backlinks |
| `get_agenda` | `filter?`, `date_range?` | Returns tasks grouped by overdue/today/upcoming. Filters: tag, status, page. |
| `toggle_task` | `title`, `task_text` (fuzzy matched) | Toggles `- [ ]` ↔ `- [x]`. Error if task not found. |
| `create_link` | `from_title`, `to_title`, `position?` | Bloom resolves titles internally. Default position: end of file. |
| `resolve_journal` | `date?` | Returns title/ID of journal page for given date (default: today). Use to target journals in other tools. |

### G18: Note Refactoring Operations

- **Split page**: Extract a section (heading + content) into a new page. The original section is replaced with a link to the new page. All existing links to blocks within that section are updated.
- **Merge pages**: Combine two pages into one. All links to the merged-away page are redirected to the target. Content is appended under a heading.
- **Move block**: Move a block from one page to another. The block's UUID follows it. All links pointing to that block are automatically valid (UUID-based, not position-based).

### G19: Templates

Bloom ships with default templates for common note types and supports user-created templates.

- **Built-in templates**: Daily journal, meeting notes, book review, project page.
- **User templates**: Stored in `templates/` as Markdown files with placeholder syntax.
- **Template selection**: When creating a new page (`SPC p n`), the picker offers available templates.
- Templates can include pre-filled frontmatter, headings, tags, and placeholder text. Placeholder syntax uses `${N:description}` (snippet-style tab-stops): `${1:Title}`, `${2:Attendees}`, etc. Magic variables `${AUTO}` (UUID), `${DATE}` (today), and `${TITLE}` (user-entered title) are filled automatically. `$0` marks the final cursor position. On template creation, the cursor jumps to `$1`, then `Tab` advances through stops. After the last stop, `Tab` exits template mode. Escape returns to Normal mode without ending template mode. If `${1:desc}` appears multiple times, advancing past it replaces all occurrences. Placeholders inside code blocks are not expanded. See [FILE_FORMAT.md](FILE_FORMAT.md) for full syntax reference.

### G20: Orphaned Link Indicators

When a page is open, Bloom validates all links against the index and shows **subtle UI hints** for broken links (deleted page, deleted block, corrupted UUID). Hints are non-intrusive — e.g., a dimmed or strikethrough link with a tooltip explaining what's broken. The link text remains visible so the user can decide how to fix it.

### G21: Setup Wizard and Vault Structure

On first launch, Bloom runs a setup wizard:

- Choose vault location (default: `~/bloom/`)
- Optionally import from Logseq (G13)

**Vault directory structure:**

```
~/bloom/                    # vault root (all of Bloom in one directory)
├── pages/                  # topic pages
│   └── Text Editor Theory.md
├── journal/                # daily journal pages
│   └── 2026-02-28.md
├── templates/              # user + built-in templates
│   ├── daily-journal.md
│   └── meeting-notes.md
├── images/                 # linked images and assets
│   └── diagram.png
├── .index/                 # SQLite index (auto-generated, rebuildable)
│   └── bloom.db
├── .gitignore              # auto-generated: excludes .index/, OS metadata
└── config.toml             # user settings, keybindings, MCP toggle
```

Backing up notes = copy `~/bloom/`. Version control = `git init` inside `~/bloom/`. Simple.
On vault creation, Bloom auto-generates a `.gitignore` that excludes `.index/` (rebuildable) and any OS metadata files. Running `git init` in the vault immediately works correctly.

**File adoption:** If a `.md` file without frontmatter appears in `pages/` or `journal/`, Bloom auto-adds frontmatter (generates UUID, derives title from filename) and shows a notification. Non-`.md` files outside `images/` are ignored.

**Git merge conflicts:** If Bloom detects git merge conflict markers (`<<<<<<<`, `=======`, `>>>>>>>`) in a file, it opens the file in raw/degraded mode (no link resolution, no indexing) and shows a warning. The user resolves conflicts manually, then Bloom re-parses and re-indexes the file on save.

### G22: Index Rebuild

`:rebuild-index` command (also available via `SPC SPC`) re-scans all files and rebuilds the SQLite index from scratch. Covers recovery from corruption, or after bulk external edits.

### G23: Session Restore

On quit, Bloom persists the session state: open buffers, window layout, cursor positions, and scroll positions. On next launch, the session is restored. Configurable in `config.toml`:

```toml
[startup]
mode = "restore"  # "restore" (default), "journal" (open today's journal), "blank"
```

Undo tree IS persisted to SQLite (G9) — restored alongside workspace layout.

### G24: Full Unicode Support

All text operations — editing, display, alignment, search, motions — must handle the full Unicode range correctly. Specifically:

- **Display width**: Column calculations use `unicode-width` display columns, not byte counts. CJK characters (2 columns), emoji, combining marks (0 columns), and multi-byte characters are handled everywhere: editor content, status bar, picker rows, alignment engine, which-key drawer.
- **Char vs byte boundaries**: String slicing always respects UTF-8 boundaries. Cursor positions, text object ranges, and search results use char indices from ropey, never raw byte offsets on user content.
- **International content**: Page titles, tags (`#café`), link display text, and frontmatter values can contain any Unicode characters. The parser uses `char::is_alphabetic()` and `char::is_whitespace()` — never ASCII-only checks — for word boundaries and tag detection.
- **Ligatures**: Font ligatures (fi, fl, →, ≥) are a rendering concern handled by the font, not the editor. Monospace fonts keep each character at a fixed column width regardless of ligature. The GUI may use ligature-capable fonts.
- **File paths**: Vault paths, filenames, and journal dates support Unicode characters on all platforms.

## Non-Goals

### NG1: Cloud / Sync

No cloud storage, no sync service, no collaboration features. This is a local-only tool. If sync is ever considered, it would be local-network / peer-to-peer — never cloud.

### NG2: Mobile Support

Desktop only (macOS + Windows). No iOS, Android, or web version.

### NG3: Plugin / Extension System (v1)

Not in scope for the initial version. However, the internal architecture is designed with future extensibility in mind — all built-in features (picker, which-key, timeline) are built on the same `BloomAPI` trait surface that plugins would eventually use.

**Future plugin vision (post-v1):**

| Layer | Mechanism | Use case |
|-------|-----------|----------|
| Core API | Rust traits (`DocumentParser`, `NoteStore`, `KeyMapper`) | Format parsers, storage backends (compile-time) |
| Lua scripting | `mlua` crate (Neovim model) | Commands, keybindings, picker sources, UI customization, workflows |
| Hooks | External process spawn | Linters, formatters, exporters, shell integrations |

The Lua layer would expose the full `BloomAPI` — buffer access, picker, commands, keybindings, search, notifications, and status bar — giving plugins the same power as built-in features.

### NG4: WYSIWYG Editing

Bloom is a markup editor with rendering, not a WYSIWYG editor. You write Markdown; Bloom renders it. Modal editing is the primary interface.

### NG5: Collaborative Editing

Single-user only. No real-time collaboration, no shared documents, no conflict resolution.

### NG6: Rich Media Embedding

No support for embedding videos, audio, web pages, or PDFs inline. Images (local files) are supported. Everything else is out of scope.

### NG7: Export

No export to PDF, HTML, or other formats. Notes are Markdown files — use external tools (Pandoc, etc.) if needed. May be revisited post-v1.

### NG8: Graph Visualization

No visual node-and-edge graph of connections. The timeline view, backlinks, and unlinked mentions provide sufficient navigability. A graph view is more gimmick than utility for most workflows.

### NG9: Multiple Vaults (v1)

Single vault only for v1. Trivial to add later since each vault is a self-contained directory — just point Bloom to a different path. No cross-vault linking.
